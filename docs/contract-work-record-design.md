# 契約管理・稼働記録の設計メモ

このドキュメントは [要件定義書](./requirements.md) のうち、`contract_party_type` と `WorkRecord` 修正履歴の実装方針を具体化した補助設計メモです。初期版で迷いやすいデータモデル上の判断を先に固定し、実装時の解釈ぶれを防ぐことを目的とします。

---

## 1. 対象範囲

このメモでは次の2点だけを扱います。

1. `contracts` テーブルに持たせる `contract_party_type` の意味と使い方
2. `work_records` の修正履歴をどのテーブルで、どの粒度で保持するか

請求計算、月次稼働報告のステータス遷移、注文書PDFの保存方式などは別途の設計対象とします。

---

## 2. 契約管理の実装方針

### 2.1 `Contract` の役割

Tulpa の `Contract` は、法律上の契約書ファイルだけを表すモデルではありません。案件に紐付く契約条件を、稼働記録・稼働報告・支払通知・請求と連動させるための業務データです。

初期版では `Contract` を「特定のエンジニアが特定の案件で稼働するための契約条件レコード」として扱います。1つの `Project` に対して複数のエンジニアが参加する場合は、エンジニアごとに `Contract` を持てる前提にします。

### 2.2 `contract_party_type` を持たせる理由

要件書では、契約は「案件に紐付く業務委託契約情報」であり、契約当事者の組み合わせを含むと定義しています。これは、Tulpa の運用上は同じ案件・同じエンジニアに紐付く契約情報でも、誰と誰の契約条件なのかで意味が変わるためです。

例:

- エージェントとクライアントの契約条件
- エージェントとエンジニアの契約条件
- エンジニアとクライアントの直接契約条件

この違いを区別せずに `contracts` を1種類で扱うと、単価・支払サイト・注文書の意味が曖昧になります。`contract_party_type` はその曖昧さを避けるための列です。

### 2.3 初期版で採用する値

初期版では `contract_party_type` を自由入力にせず、アプリケーション側の Enum で次の3値に固定します。

| 値 | 意味 | 主な用途 |
|------|------|------|
| `agent_client` | エージェント-クライアント間の契約条件 | 請求書発行、クライアント向け条件確認 |
| `agent_engineer` | エージェント-エンジニア間の契約条件 | 支払通知、エンジニア向け条件確認 |
| `client_engineer_direct` | クライアント-エンジニア間の直接契約条件 | 直請け案件の条件管理 |

`other` のような曖昧な値は初期版では入れません。必要になった時点で要件とUIを見直して追加します。

### 2.4 レコードの持ち方

同じ `project_id` と `engineer_id` に対して、`contract_party_type` の異なる `Contract` を複数持てるようにします。つまり、1人のエンジニアが1つの案件に入るとき、必要に応じて次の2レコードを併存させます。

- `agent_client`
- `agent_engineer`

この構成により、クライアントへの請求条件とエンジニアへの支払条件を分離できます。逆に、直請け案件では `client_engineer_direct` のみを持つ想定です。

初期版では、`contracts` テーブルに対して次の一意制約を置く方針とします。

- `project_id`
- `engineer_id`
- `contract_party_type`
- `start_date`

これにより、同一期間に同じ種類の契約条件を重複登録しにくくします。

### 2.5 他機能との関連

- `WorkRecord` と `WorkReport` は、稼働記録・稼働報告の入力主体であるエンジニア側の契約条件に連動させるため、原則として `agent_engineer` または `client_engineer_direct` の `Contract` を参照します。
- `Invoice` はクライアント向け請求なので、原則として `agent_client` の条件を参照します。
- `PaymentNotice` はエンジニア向け支払なので、原則として `agent_engineer` または `client_engineer_direct` を参照します。

将来的に「1つの契約セット」としてまとめて扱いたくなった場合でも、初期版はまず各レコードの意味を明確に保つことを優先します。

## 3. 稼働記録修正履歴の実装方針

### 3.1 `WorkRecord` 本体に履歴を埋め込まない理由

要件書では、稼働記録の修正時に履歴を残すことを必須にしています。これを `work_records` テーブルの列追加だけで表現すると、最新値は持てても、何がどう変わったかを複数回追跡しづらくなります。

そのため初期版では、最新の稼働記録は `work_records` に保持し、修正履歴は別テーブルに追記する方式を採用します。

### 3.2 採用するテーブル

初期版では `work_record_revisions` テーブルを追加し、1回の修正につき1行を保存します。

| カラム | 内容 |
|------|------|
| `id` | 主キー |
| `work_record_id` | 対象の稼働記録 |
| `changed_by_user_id` | 修正を行ったユーザー |
| `previous_start_time` | 修正前の作業開始時刻 |
| `previous_end_time` | 修正前の作業終了時刻 |
| `new_start_time` | 修正後の作業開始時刻 |
| `new_end_time` | 修正後の作業終了時刻 |
| `reason` | 修正理由 |
| `created_at` | 修正日時 |

初期版では、修正理由を必須入力とします。自由記述でよく、コード化は行いません。

### 3.3 履歴を残すタイミング

履歴を残すのは、既存の `WorkRecord` が更新されたときだけです。最初の登録時点では履歴を作りません。

具体的なルールは次のとおりです。

- 新規作成: `work_records` のみ作成する
- 時刻変更: 変更前後の値を `work_record_revisions` に保存してから `work_records` を更新する
- 理由のみ変更: 初期版では想定しない。修正理由は履歴とセットで残す

削除操作は初期版では提供しない前提にします。不要なレコードの扱いが必要になった場合は、削除ではなく無効化または差戻しフロー側で吸収する設計を検討します。

### 3.4 実装時の更新順序

整合性を保つため、`WorkRecord` の更新はトランザクション内で次の順序で行います。

1. 現在の `work_records` の値を読み出す
2. `work_record_revisions` に変更前後の値と修正理由を保存する
3. `work_records` の最新値を更新する

この順序により、途中失敗で履歴だけ残る状態を防ぎます。

### 3.5 表示方針

初期版のUIでは、最新の稼働記録詳細から修正履歴を時系列で確認できる状態にします。表示項目は次に絞ります。

- 修正日時
- 修正者
- 修正前の作業開始・作業終了
- 修正後の作業開始・作業終了
- 修正理由

差分表示はこの粒度で十分とし、初期版では監査ログ専用画面までは作りません。

## 4. Laravel 実装メモ

### 4.1 Enum

`contract_party_type` は PHP Enum で表現します。

想定例:

```php
enum ContractPartyType: string
{
    case AgentClient = 'agent_client';
    case AgentEngineer = 'agent_engineer';
    case ClientEngineerDirect = 'client_engineer_direct';
}
```

### 4.2 モデル分離

- `Contract` モデルは契約条件の本体を持つ
- `WorkRecord` モデルは最新の稼働記録を持つ
- `WorkRecordRevision` モデルは修正履歴だけを持つ

Eloquent の Observer は使わず、修正履歴の記録はサービスクラスで明示的に行います。暗黙の副作用を避けるためです。

### 4.3 サービス責務

初期版では少なくとも次の責務をサービスクラスに寄せます。

- `ContractService`: `contract_party_type` を含む契約登録・更新の検証
- `WorkRecordService`: 稼働記録の新規作成、修正履歴保存を伴う更新

---

## 5. 受け入れ条件

このメモに基づく実装が完了したと判断する条件は次のとおりです。

1. 同じ案件・同じエンジニアに対して `agent_client` と `agent_engineer` の契約条件を区別して保存できる。
2. 稼働記録を修正すると、修正前後の時刻と修正理由が履歴として残る。
3. 月次稼働報告と支払通知が、参照すべき契約条件を `contract_party_type` で判別できる。
4. 実装コード中で `打刻` ではなく `作業開始` / `作業終了` / `稼働記録` の用語を使う。
