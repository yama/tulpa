# Tulpa 要件定義書

**Version 2.0**
**2026年2月28日**

---

## 1. プロジェクト概要・設計思想

### 1.1 背景と問題意識

#### エンジニアと担当者のすれ違い

エンジニアは「自分はこんなことができる・こんな仕事がしたい」という思いを持っている。しかし担当者にはIT技術のバックグラウンドがないことも多く、スキルシートの内容を十分に理解できないまま、表面的な条件だけでマッチングが行われることがある。「自分の本当の強みが担当者に伝わっていない」というジレンマは、多くのエンジニアが抱えている。

担当者側にも共通の課題がある。スキルシートを深く理解できないまま、自信を持ってエンジニアをクライアントに推薦しなければならない場面は、エージェントの規模を問わず生じる。理解しきれていないことへの後ろめたさを抱えながら、提案は続く。

#### 共通する構造的問題

- 稼働・請求・契約の情報が別々のツールに散在し、案件単位で繋がっていない
- コミュニケーションがメール・電話・LINEに分散し、案件のコンテキストと切り離されている
- マッチングは担当者の「勘」に頼った手作業で行われており、見落としが生じやすい
- 担当者によって対応品質に大きなばらつきがある

これらは個別の問題ではなく、**「案件を中心にデータが繋がっていない」という一つの構造的問題**から来ている。

### 1.2 Tulpaとは

Tulpaは、**フリーランスエージェント会社の事務業務をシステム化するオープンソースの業務管理ツール**である。稼働記録・契約管理・支払通知を中心に、これまで属人的な運用や老朽化したシステムに頼っていた業務を整理・自動化する。担当者が事務作業から解放されることで、エンジニアとクライアントを繋ぐ本来の仕事に集中できる環境をつくる。

加えて、エンジニアが「こういう働き方がしたい」という意思を担当者に届けやすくする仕組みを備えている。AIがスキルシートを読み解いてわかりやすく翻訳することで、担当者がエンジニアの強みを理解しやすくなる。こうしたAI機能はオプションであり、コア機能はAI未設定でもすべて利用できる。

---

#### Tulpaという名前について

Tulpaはチベット語由来の言葉で、「思念が形になる」を意味する。担当者・エンジニア・クライアント、それぞれが持つ「こういう人に来てほしい」「自分はこんなことができる」「このエンジニアと合うはずだ」という思いは、データとして散在したままでは形にならない。案件を軸にデータが繋がることで、その思いが自然に形になっていく——という設計思想を、この名前に込めている。

### 1.3 設計思想

**1. 案件を中心にデータを繋げる**
勤怠・請求・契約・コミュニケーションは、すべて「案件・クライアント」という軸で紐付ける。O社のアイコンをタップしたら、O社に関するすべての情報が一つの文脈として現れる。機能別の画面遷移ではなく、案件中心のナビゲーションを基本とする。

**2. ミニマムな見せ方にこだわる**
情報は豊富にあるが、一度に見せる量は最小限にする。ログイン直後は「今日やること」だけが見える。詳細は必要なときに必要な深さで現れる。「シンプルで簡単そう」という第一印象を最優先する。

**3. OSSとして公開する**
特定のエージェント会社に依存せず、誰でも導入・改善できるOSSとして公開する。レンタルサーバーでも動作する軽量設計を維持する。

**4. エンジニアの努力を、担当者が発見できるようにする**
IT業界の変化は速い。しかしエージェント会社の担当者がその変化についていくことは難しく、毎日コードを書き、記事を書き、勉強会に参加しているエンジニアの努力は、担当者の目に届きにくい。担当者がエンジニアの技術的な成長に気づけなければ、クライアントへの推薦は表面的なものになってしまう。

担当者がエンジニアの価値を理解する入口は、スキルシートをAIが読み解くことにある。プログラミング経験のない担当者には、スキルシートの技術的な内容の意味が伝わりにくい。AIが「このエンジニアの強みはこういうことです」と言語化するだけで担当者の理解が深まり、推薦の質は変わる。外部サービスのデータ（GitHub・Qiita・WakaTimeなど）はその理解を補強する任意の証跡だが、これらを使っていない優れたエンジニアも多い。スキルシート単体で十分な価値を提供することが前提であり、外部連携はオプションである。使い続けることで、担当者自身が業界の変化に追いつく学びの場にもなる。

**5. 自動化は手段、気づきと学びが本質**
Tulpaが提供する効率化・自動化は、それ自体が目的ではない。担当者がエンジニアを深く理解し、クライアントへの推薦の質が変わること——その先にある「気づき」と「学び」こそが、このシステムの本質的な価値である。「AIを使う」という操作を求めない。スキルシートと案件情報を登録するだけで学びが生まれ、使い続けることで担当者自身も成長する設計にする。ただし、これを前面に打ち出すと「情報分析系ツール」と誤解され、本来届けたい担当者に届かなくなる。事務管理という入口から自然に使い始め、使い続けることでその真価が現れる——この順序が設計の意図である。

### 1.4 このシステムが変えること

| 現状 | Tulpa |
|------|-------|
| 月末にまとめて稼働記録を入力・Excel提出 | その場でスマホ入力、そのまま提出 |
| 請求書・支払通知を毎月手作業で作成 | 稼働確認後に自動生成 |
| 契約更新を見落とすリスク | 期限前に自動アラート通知 |
| 情報が機能別ツールに分散 | 案件・クライアント中心に統合 |
| メール・電話・LINEに分散 | Slackをハブに自然移行（オプション） |
| 担当者の「勘」に頼ったマッチングで見落としが生じやすい | 担当者の勘をAIがデータで補強（オプション） |
| 担当者によって対応品質に大きなばらつき | AIが客観的なベースラインを提供し、担当者が学びながら質を高められる |

### 1.5 想定利用者

- 小規模フリーランスエージェント会社の担当者
- エージェント的な役割を担うフリーランサー個人
- エンジニアコミュニティで自律的に導入・運用するグループ

## 2. 用語定義

### 2.1 設計思想としての言葉遣い

フリーランスエンジニアは業務委託契約に基づく個人事業者であり、時間・場所による指揮命令関係は発生しない。システムの言葉遣いやUI設計が「雇用関係」を想起させないよう、以下の用語を統一して使用する。

| 使用しない表現 | Tulpaでの表現 | 理由 |
|-------------|-------------|------|
| 勤怠管理 | 稼働記録 | 「勤怠」は雇用関係を前提とした言葉 |
| 打刻・出退勤 | 作業開始・作業終了 | 出勤・退勤の概念を使わない |
| 勤務時間 | 稼働時間 | 同上 |
| 承認（クライアントによる） | 確認 | クライアントは指揮命令者ではない |
| 上長・責任者 | クライアント担当者 | 指揮命令関係を示唆しない |
| 遅刻・早退 | （使用しない） | 業務委託契約では不適切な概念 |
| 勤怠承認ワークフロー | 稼働確認フロー | 同上 |
| 作業報告書 | 稼働報告 | より中立的な表現 |

### 2.2 システム内の主要用語

| 用語 | 定義 |
|------|------|
| 稼働記録 | エンジニアが自律的に記録する作業時間のログ |
| 稼働確認 | クライアント担当者による稼働記録の確認行為 |
| 案件 | エンジニアが業務委託を受ける仕事の単位 |
| 契約 | エンジニアとクライアント企業間の業務委託契約 |
| 稼働報告 | 月次の稼働時間をまとめてクライアントに共有する行為 |
| スキルシート | エンジニアの技術スキル・経験をまとめたプロフィール |
| マッチング | スキルシートと案件要件をAIが照合するプロセス |

## 3. ユーザーと権限

### 3.1 ユーザー種別

| 種別 | 説明 | 主な関心事 |
|------|------|-----------|
| オーナー | システム全体の管理者（エージェント会社の代表等） | テナント設定、全データの閲覧 |
| エージェント | 案件・契約・マッチングを担当する担当者 | 案件管理、エンジニア管理、承認 |
| エンジニア | フリーランスエンジニア本人 | 稼働記録、契約・支払確認 |
| クライアント担当者 | クライアント企業側の担当者 | 稼働確認（将来：稼働状況の確認） |

### 3.2 権限設計の方針

v1.1の「システム管理者」という概念を廃止し、**オーナー**に統合する。初期はシングルテナント（1エージェント会社）を前提とし、マルチテナント対応は将来課題とする。

権限はロールベースで管理するが、複雑な権限マトリクスは避ける。「エージェントはすべての案件・契約を操作できる」「エンジニアは自分の情報だけ見える」「クライアント担当者は自社に関連する情報だけ見える」というシンプルな原則を基本とする。

### 3.3 各ユーザーの主な操作範囲

**オーナー**

- エージェントアカウントの作成・管理
- システム設定（Slack連携、メール設定等）
- 全データの閲覧・エクスポート

**エージェント**

- クライアント・案件の登録・編集
- エンジニアの招待・スキルシート管理
- 稼働の確認・差戻し
- 支払通知書・請求書の発行
- AIマッチングサマリーの生成・編集

**エンジニア**

- 稼働記録・月次提出
- 自分の契約・支払通知の確認
- スキルシートの編集
- 案件の閲覧・興味表明

**クライアント担当者**

- 担当案件のエンジニアが提出した稼働記録の確認・差戻し
- （将来）稼働状況・契約情報のポータル閲覧

### 3.4 クライアント企業の担当者管理

クライアント企業には複数の担当者が存在し、異動・退職等による担当者交代も発生する。以下の点を考慮して設計する。

- クライアント企業（組織）とクライアント担当者（個人）を分離して管理する
- 1社に対して複数の担当者アカウントを紐付けられる
- 担当者が交代した場合、旧担当者のアカウントを無効化し、承認履歴は保持する
- 担当者の招待はエージェントがメールで行う（クライアント自身がアカウント作成）
- 担当者が見えるのは自社案件の情報のみに限定する

## 4. コア機能

### 4.1 稼働記録

#### 4.1.1 設計方針

**作業の開始・終了時にスマホからその場で記録する**方式を推奨とする。ただし月末まとめ入力も引き続き利用できるフォールバックとして維持する。毎日の打刻を強制しない。エンジニアが自律的に記録するという業務委託契約の性質を、UIの設計にも反映させる。

#### 4.1.2 作業開始・終了の記録

- スマホのホーム画面から1タップで作業開始・終了を記録できる
- 記録には案件を紐付ける。案件が1つの場合は選択画面を省略し即時記録する。複数案件を掛け持ちしている場合のみ選択画面を表示する
- 記録時刻は自動入力、後から修正も可能
- 修正した場合は修正履歴を残す（透明性の確保）
- オフライン時はローカルに保持し、オンライン復帰時に同期する

#### 4.1.3 稼働記録の確認・編集

- 日次・週次・月次で稼働時間のサマリーを確認できる
- 記録漏れがあった場合は後から手動入力できる（月末まとめ入力もここで対応）
- 休憩時間の記録は任意（精算幅の計算に影響する場合のみ）
- 土日・祝日は視覚的に区別するが、稼働記録自体は制限しない

#### 4.1.4 月次稼働報告

- 月次の稼働記録を一覧で確認し、クライアントに共有する
- 共有前にエンジニア自身が内容を確認・修正できる
- 共有後はクライアント担当者が確認（差戻し可能）
- 差戻し時はコメントとともに通知される
- 確認済みになった稼働報告は請求処理に連動する

#### 4.1.5 現場指定フォーマットへの対応

一部のクライアントは独自の稼働報告書フォーマットを指定する場合がある。その場合はファイルアップロードで対応する。

- PDFまたはExcelファイルのアップロード
- アップロード後、合計稼働時間を手動入力して共有
- ファイルはクライアント担当者が確認できる状態で保存

---

### 4.2 契約管理

#### 4.2.1 設計方針

契約情報は「管理される書類」ではなく、**案件・稼働・支払のすべてが紐付く中心的なデータ**として設計する。エンジニアもエージェントもクライアントも、契約を起点に関連情報へアクセスできる。

#### 4.2.2 契約の登録・管理（エージェント）

- クライアント企業・案件・エンジニアを紐付けて契約を登録する
- 契約期間（開始日・終了日）、単価、精算幅、支払いサイトを管理する
- 注文書PDFのアップロード・保管
- 契約更新・終了が近づいた場合にエージェントへ通知する（30日前・7日前）
- 契約履歴を時系列で保持する（更新・終了の経緯が追える）

#### 4.2.3 契約の確認（エンジニア）

- 現在有効な契約の概要がダッシュボードで確認できる
- 契約期間・単価・精算幅・支払いサイトを確認できる
- 次回更新日・終了日が明示される
- 過去の契約履歴も閲覧できる

#### 4.2.4 契約と他機能の連動

- 稼働記録は契約に紐付いて蓄積される
- 月次稼働報告は契約の精算幅をもとに過不足を自動計算する
- 月の途中でも現在の稼働時間と月末見込みを表示し、精算幅に対する進捗をリアルタイムで確認できる（エンジニアが自律的にペース調整できるよう支援する）
- 年間カレンダービューで月ごとの営業日数・標準稼働時間（祝日考慮済み）を表示する。精算幅に対して何日休めるかの余裕日数も示し、エンジニアが年間を通じた休暇計画を立てられるようにする（詳細は画面設計セクション参照）
- 支払通知は契約の支払いサイトをもとに生成される
- 契約が終了した場合、稼働記録の入力は自動的に締め切られる

---

### 4.3 支払通知・請求

#### 4.3.1 設計方針

支払通知と請求書は「PDFを発行して終わり」ではなく、**契約・稼働報告と連動して自動生成される**ものとして設計する。エンジニアにとっては「いくら・いつ振り込まれるか」が一目でわかることを優先する。

#### 4.3.2 支払通知（エンジニア向け）

- 月次稼働報告が確認済みになると、支払通知が自動生成される
- 支払金額・支払日・対象期間が明示される
- 精算幅の超過・不足がある場合は、単価計算の内訳も表示する
- インボイス制度対応：適格請求書発行事業者番号の管理・表示
- 支払通知PDFのダウンロード
- 支払日が近づいたら通知する（Slack・メール）

#### 4.3.3 請求書発行（エージェント向け）

- 確認済みの稼働報告をもとに請求書を自動生成する
- クライアント企業ごとに複数エンジニアの請求をまとめて発行できる
- 請求書PDFの発行・送付履歴の管理
- 支払いサイトに基づく入金予定日の管理
- 未入金の場合はアラート通知

#### 4.3.4 振込先・支払情報の管理

- エンジニアは振込先銀行口座を登録・管理できる
- 屋号・適格請求書発行事業者番号（インボイス登録番号）を管理する
- 登録番号は請求書・支払通知に自動反映される。消費税計算の詳細はfreee・マネーフォワード等の会計ソフトに委ねる割り切りとし、Tulpa側では複雑なインボイス計算は行わない
- 情報変更時はエージェントに通知される

---

### 4.4 案件管理・AIマッチング

#### 4.4.1 設計方針

案件管理の核心は、**スキルシートと案件要件をAIが照合し、マッチング度と理由を自動生成する**ことにある。エージェントは「この案件に合うエンジニアは誰か」をワンクリックで確認でき、エンジニアは「自分のスキルに合う案件はどれか」を一覧で確認できる。生成されたサマリーはエージェントが編集・補足してから客先に提出する。AI生成であることは画面上に明示する（透明性の確保）。

#### 4.4.2 案件登録（エージェント）

- 案件名・概要・必要スキル・単価レンジ・勤務地・リモート可否・稼働率を登録する
- 必要スキルにはAI活用能力の項目を明示的に設ける（例：ChatGPT活用、GitHub Copilot、プロンプトエンジニアリング等）
- 募集期間・商流・公開/非公開設定
- クライアント企業との紐付け

#### 4.4.3 スキルシート管理（エンジニア）

- 技術スキル・経験年数・得意領域をエンジニア自身が登録・更新する
- AIツール活用経験を独立した項目として設ける（使用ツール・活用レベル・具体的な活用事例）
- 過去の稼働実績は契約データから自動的に蓄積される
- スキルシートはPDF出力できる（客先提出用）

#### 4.4.4 候補者サマリーの自動生成

現状、エージェントが担当者の「勘」を頼りに手作業で作成している「こんなエンジニアがいます」という客先向けサマリーを、AIが下書きとして自動生成する。担当者は生成内容を確認・編集し、自分の勘や文脈を加えて仕上げる。

- エンジニアのスキルシート・稼働実績をもとにサマリー文を生成する
- 案件要件に合わせてサマリーの内容を調整できる（案件特化型サマリー）
- エージェントが編集・承認してから客先に送付する
- 過去に送付したサマリーの履歴を保持する

#### 4.4.5 案件閲覧・興味表明（エンジニア）

- 公開案件の一覧をスキルでフィルタリングして閲覧できる
- 気になる案件にブックマーク・興味表明ができる
- 応募履歴・選考状況を確認できる
- AIが「あなたのスキルに合いそうな案件」をサジェストする

### 4.5 Slack連携（オプション）

#### 4.5.1 設計方針

Slack連携はオプション機能として提供する。未設定でもTulpaのコア機能はすべて利用できる。Slackを導入済みのエージェントが設定することで、コミュニケーションが案件のコンテキストと紐付いて蓄積される。

Slackへの移行を強制するものではなく、**Slackを使い始めたくなる環境を整える**という位置づけにする。

#### 4.5.2 通知連携

- 稼働報告の提出・確認・差戻しをSlackに通知する
- 契約更新アラートをSlackに通知する
- 支払通知の発行をSlackに通知する
- 通知先チャンネルはエージェントが設定できる

#### 4.5.3 コミュニケーションの記録

Slackのメッセージを案件に自動紐付けすることは、実運用上難しい（やり取りが複数チャンネルに混在するため）。無理に自動化せず、以下の範囲に留める。

- TulpaからSlackへの通知は案件・エンジニアを明示したフォーマットで送信する（受信側がO社関連と判断しやすくする）
- 重要なやり取りをエージェントが手動でTulpaにメモとして記録できる
- ハドル（音声通話）の実施記録を手動で残せる（日時・参加者・メモ）

---

### 4.6 エンジニアアクティビティ連携・公開プロフィール

#### 4.6.1 設計方針

スキルシートに書かれた「得意なこと」は自己申告にすぎない。しかし、その内容を正確に理解できる担当者は多くない。技術的な経歴や得意領域が文字として書かれていても、プログラミング経験のない担当者にはその意味が伝わりにくい。**AIがスキルシートを読み解き、担当者の言葉に翻訳するだけでも、発見の質は大きく変わる。**

外部サービスのデータ（GitHubのコントリビューション・Qiitaの記事・WakaTimeのコーディング時間など）は、スキルシートへの理解をさらに補強する**任意の証跡**として位置づける。GitHubやQiitaを使っていない優れたエンジニアも多く、外部サービスを使っているかどうか自体が評価に影響してはならない。連携はあくまで、使っているエンジニアをより深く理解するためのオプションである。

この機能の本質的な価値は、 **エージェント担当者の「発見」** にある。担当者がエンジニアの詳細画面を開いたとき、「このエンジニア、こんなことに取り組んでいたのか」という気づきが生まれる設計にする。その発見が、クライアントへの推薦の質を変える。

これらを集約した公開プロフィールページを `tulpa.example.com/p/{slug}` の形式で発行する。エンジニアにとっては「キャリアポートフォリオ」として機能し、エージェントにとってはクライアントへの提案URLとして機能する。LAPRASがこのページの存在自体を差別化要因にしているように、Tulpaもこの機能をエンジニアが使い続ける理由にする。

#### 4.6.2 連携する外部サービス（いずれも任意）

| サービス | 取得データ |
| ------- | --------- |
| WakaTime | コーディング時間・使用言語分布・週次推移 |
| GitHub | コントリビューショングラフ・主要言語・リポジトリ情報 |
| Qiita / Zenn | 記事数・総いいね数・タグ傾向・最新記事 |
| Connpass | 参加イベント数・登壇歴（特に強調表示） |

#### 4.6.3 公開プロフィールページ

- `tulpa.example.com/p/yamamoto` のような固定URLで公開できる
- エンジニアが公開する項目を個別にON/OFFできる（連携データ・スキル・稼働実績の有無等）
- エージェントはクライアント担当者への提案時にこのURLをシェアできる（候補者サマリーにURLを添付）
- 「AIが生成したプロフィールサマリー文」をページに組み込める（エンジニアが選択・編集）
- ページ閲覧数・URLシェア数をエンジニア自身が確認できる
- ログイン不要で閲覧できる（クライアント担当者がアカウントなしで確認できる）

#### 4.6.4 公開プロフィールに表示する情報

| 項目 | 表示内容 | 公開設定 |
| ------ | ---------- | ---------- |
| ヘッダー | 名前・アイコン・一言紹介・稼働ステータス | 必須 |
| スキルサマリー | タグ形式のスキルセット・AIツール活用経験 | 任意 |
| WakaTimeデータ | 言語分布グラフ・週次コーディング時間 | 任意 |
| GitHubアクティビティ | コントリビューショングラフ・主要言語 | 任意 |
| Qiita / Zenn記事 | 記事一覧・総いいね数 | 任意 |
| Connpassイベント | 参加数・登壇履歴 | 任意 |
| AIプロフィール文 | AI生成・エンジニア編集済みの自己紹介 | 任意 |
| 稼働実績 | 「〇年の稼働実績あり」程度の抽象表現のみ | 任意 |

クライアント名・単価・具体的な契約情報は公開プロフィールには一切表示しない。

#### 4.6.5 データ収集とプライバシー

- 各サービスのAPIを使用してデータを定期取得（日次バッチ）。エンジニアが連携設定でAPIトークン・ユーザー名を登録する
- Tulpa側ではデータをキャッシュし、API呼び出しを最小化する
- 連携を解除した場合、取得済みデータの扱い（保持 or 削除）はエンジニアが選択できる
- デフォルトは非公開。エンジニアが明示的に公開に切り替える
- 稼働ステータスは「稼働中 / 案件相談可 / 休止中」の3択のみ公開可能
- 取得データはエージェント画面のエンジニア詳細・候補者サマリー生成時のAI入力としても活用する

## 5. 画面設計

### 5.1 設計方針

「シンプルで簡単そう」という第一印象を最優先する。ログイン直後に情報を詰め込まず、今必要なことだけが見える。詳細は案件・クライアントを起点に掘り下げていく構造にする。

スマホファーストで設計し、PCでも違和感なく使えるレスポンシブ対応とする。

### 5.2 エンジニア向け画面

#### 5.2.1 ダッシュボード（ログイン直後）

今日必要なことだけを表示する。情報過多にしない。

- **稼働状況カード**：今月の稼働時間 / 精算幅に対する進捗バー / 月末見込み時間
- **作業開始・終了ボタン**：画面中央に大きく配置。案件が1つなら即時記録、複数なら選択画面へ
- **アクションが必要なもの**：差戻しがあれば表示、それ以外は何も出さない

#### 5.2.2 年間カレンダービュー

- 12ヶ月を一覧で表示
- 月ごとの営業日数・標準稼働時間（祝日考慮済み）を表示
- 精算幅の下限に対して「あと何日休めるか」を月ごとに表示
- 稼働記録済みの日は視覚的にマーク
- 2月のような短い月や連休が多い月が一目でわかる

#### 5.2.3 案件・契約詳細

- 契約期間・単価・精算幅・支払いサイト・次回更新日
- 月次稼働報告の一覧（ステータス付き）
- 支払通知の履歴
- エージェントとのやり取りメモ
- 関連するSlack通知の履歴（オプション）

#### 5.2.4 スキルシート

- 技術スキル・経験年数の編集
- AIツール活用経験の編集
- PDF出力ボタン
- 案件マッチングのサジェスト（「あなたのスキルに合う案件」）
- 外部サービス連携設定（WakaTime・GitHub・Qiita・Zenn・Connpass）
- 公開プロフィールの設定（公開ON/OFF・各セクションの表示切替・公開URL確認）

#### 5.2.5 公開プロフィールページ

ログイン不要で閲覧できる外部公開ページ。表示項目は4.6.4の定義に従い、エンジニアがセクションごとにON/OFFできる。フッターにTulpaへのリンクを表示する。エージェントはエンジニア詳細画面からこのURLをコピーして、クライアントへの提案時に添付できる。

#### 5.2.6 ナビゲーション構成（エンジニア）

画面下部のタブナビゲーション（スマホ基準）。項目は最小限にする。

| タブ | 内容 |
|------|------|
| ホーム | ダッシュボード |
| 稼働 | 稼働記録・年間カレンダー |
| 案件 | 契約詳細・支払通知・案件一覧 |
| プロフィール | スキルシート・口座・設定 |

### 5.3 エージェント向け画面

#### 5.3.1 ダッシュボード

- 対応が必要なもののみ表示（稼働確認待ち・契約更新アラート・未入金）
- 何もなければ「今日は対応不要です」と表示する（空白を恐れない）

#### 5.3.2 クライアント中心ビュー

クライアントを起点に、営業活動と提案戦略を管理する。

- クライアント一覧（取引中・提案中・休眠のステータス付き）
- クライアント詳細画面：
  - 基本情報（担当者・連絡先）
  - 取引履歴（過去の案件・契約・エンジニア）
  - 属性メモ（好むスキルセット・予算感・稼働条件の傾向など）
  - 次の提案計画と期日（リマインド通知付き）
  - 過去のアプローチ履歴とその結果
- 「フォローが必要なクライアント」をダッシュボードに表示（次のアクション期日が近いもの）
- AIを活用して「このクライアントに合うエンジニア候補」をサジェストできる

#### 5.3.3 エンジニア中心ビュー

担当エンジニアを起点に、営業活動の状況を管理する。現状Excelで属人的に管理されている情報をシステム化する。

- エンジニア一覧（稼働中・営業中・待機中のステータス付き）
- エンジニア詳細画面：
  - 基本情報・スキルシート
  - アプローチ履歴（日付・クライアント名・案件名・結果・メモ）
  - 現在の選考状況（提案中・面談設定・内定・辞退・不採用）
  - 次のアクションと期日（リマインド通知付き）
  - 稼働中の場合は契約情報・稼働状況へのリンク
- 「フォローが必要なエンジニア」をダッシュボードに表示（次のアクション期日が近いもの）

#### 5.3.4 AIマッチング画面

- 案件を選択すると候補エンジニアとマッチング度・理由が表示される
- サマリー文を編集して客先送付用に仕上げる
- 送付履歴を保持する

### 5.4 クライアント担当者向け画面

MVPでは最小限の画面構成とする。

- 稼働報告一覧（確認待ちのみハイライト）
- 稼働報告詳細・確認・差戻し
- 差戻しコメント入力

## 6. 技術スタック

PHP 8.2+ / Laravel 12・Alpine.js + htmx・Tailwind CSSを採用する。レンタルサーバーでも動作する軽量設計はOSSとしての重要な要件であり、変えない。Claude API・Slack API・PWA対応を追加する。

### 6.1 技術スタック一覧

| レイヤー | 技術 | 備考 |
|---------|------|------|
| 言語 | PHP 8.2+ | 型宣言・Attributes・Enumの活用 |
| フレームワーク | Laravel 12 | ルーティング・ORM・認証・テンプレートを包括 |
| テンプレート | Blade | Laravel組み込み |
| ORM | Eloquent | Laravel組み込み |
| フロントエンド | Alpine.js + htmx | ビルド不要、CDN配信 |
| CSS | Tailwind CSS | ビルド済み軽量版を同梱 |
| データベース | MySQL 8.0+ / MariaDB 10.6+ | JSON型・CTE対応 |
| 認証 | Laravel Auth | Laravel組み込み |
| PDF生成 | TCPDF / mPDF | 支払通知・請求書・スキルシート生成 |
| AI連携 | Claude API | マッチング・サマリー生成 |
| Slack連携 | Slack Web API | 通知・メッセージ連携（オプション） |
| メール送信 | Laravel Mail | 通知・招待メール |
| PWA | Service Worker + Web App Manifest | スマホからホーム画面追加・オフライン対応 |

### 6.2 PWA対応について

スマホファーストの打刻UXを実現するため、ネイティブアプリではなくPWA（Progressive Web App）として実装する。

- ホーム画面に追加することでアプリのように起動できる
- プッシュ通知（差戻し・契約更新アラート・打刻忘れ通知等）に対応する
- 前日の打刻記録がない場合、翌朝に「昨日の稼働記録がありません」と通知する
- オフライン対応は優先度低とし、将来の拡張候補とする
- App Store / Google Playへの申請不要で配布できる（OSSとして重要）

**プッシュ通知の制約：** iOSでのPWAプッシュ通知はiOS 16.4以降・ホーム画面追加済みの場合のみ動作する。iOS Safariのブラウザタブ上では通知は受信できないため、通知機能の利用にはホーム画面追加を案内するオンボーディングを設ける。

### 6.3 Claude API連携について

- マッチング・サマリー生成に使用する
- APIキーはオーナーが設定画面から登録する（システムに同梱しない）
- 未設定の場合はAI機能が非表示になる（コア機能には影響しない）
- API呼び出しはサーバーサイドで行い、APIキーをクライアントに露出しない

### 6.4 Slack API連携について

- Slack Appをエージェントが自分のワークスペースにインストールする形式
- 設定はオーナー画面から行う（OAuth 2.0による認証）
- 未設定の場合はSlack連携機能が非表示になる（コア機能には影響しない）

## 7. データモデル

「案件・クライアントを中心にデータが繋がる」という設計思想をデータモデルでも体現する。すべての主要エンティティはClientまたはProjectに紐付く形で設計する。

### 7.2 主要エンティティ

| エンティティ | 説明 | 主要属性 |
|------------|------|---------|

| PaymentNotice | 支払通知 | contract_id, year_month, amount, pdf_path |
| Invoice | 請求書 | client_id, year_month, amount, issued_at |
| OrderDocument | 注文書 | contract_id, pdf_path |
| EngineerApproach | エンジニア営業履歴 | engineer_id, client_id, project_id, approached_at, status, memo, next_action_at |

| EngineerActivity | 外部サービス連携データ | engineer_id, service(github/wakatime等), data_json, fetched_at |
| PublicProfile | 公開プロフィール設定 | engineer_id, slug, is_public, visible_sections, profile_text |

### 7.3 エンティティ関係の概要

- **Client** を中心に Project・ClientContact・Contract・Invoice・ClientApproach が紐付く
- **Engineer** を中心に Contract・SkillSheet・WorkRecord・EngineerApproach・EngineerActivity・PublicProfile が紐付く
- **Contract** を中心に WorkRecord・WorkReport・PaymentNotice・OrderDocument が紐付く
- **WorkRecord** は修正履歴（is_modified・modified_reason）を持ち、透明性を確保する

### 7.4 v1.1からの主な変更点

- **Engineer** に ai_skills（AIスキル）を追加
- **Client** に status・next_action_at（次のアクション期日）を追加
- **ClientContact** を新設（クライアント企業と担当者を分離）
- **WorkRecord**（日次）を新設。月次まとめ入力のフォールバックも同テーブルで対応
- **EngineerApproach・ClientApproach** を新設（CRM機能）
- **AiMatchingLog** を新設（マッチング履歴の保持）
- **ActivityLog** を新設（コミュニケーションメモ）
- **EngineerActivity** を新設（WakaTime・GitHub等の外部サービス連携データキャッシュ）
- **PublicProfile** を新設（公開プロフィールの設定・スラッグ管理）

## 8. 開発スケジュール

### 8.1 基本方針

機能を一度にすべて実装しようとせず、**動くものを早く公開する**ことを優先する。OSSとして公開することで、フィードバックを得ながら段階的に改善していく。

AI機能・Slack連携はコア機能が安定してから追加する。ポートフォリオとしての観点からも、execPlanの方法論に従い開発プロセス自体を透明に記録・公開する。

### 8.2 フェーズ構成

#### Phase 1: 基盤・コア機能（1〜2ヶ月目）

目標：エンジニアが稼働記録を提出し、クライアントが確認できる状態

- 認証・認可（4ロール）
- クライアント企業・担当者管理
- エンジニア管理・スキルシート（AIスキル項目含む）
- 案件・契約管理
- 稼働記録（打刻型・手動入力）
- 月次稼働報告・確認フロー
- PWA対応（ホーム画面追加・プッシュ通知）
- 年間カレンダービュー

#### Phase 2: 業務効率化機能（3ヶ月目）

目標：エージェント業務のCRM化・請求処理の自動化

- 支払通知・請求書の自動生成
- エンジニアCRM（アプローチ履歴・次アクション管理）
- クライアントCRM（提案履歴・次アクション管理）
- コミュニケーションメモ
- 契約更新アラート・打刻忘れ通知

#### Phase 3: AI・連携機能（4ヶ月目以降）

目標：AIマッチングによる差別化・OSS公開

- Claude APIによるマッチング・サマリー生成
- Slack連携（オプション）
- スキルシートPDF出力
- 外部サービス連携（WakaTime・GitHub・Qiita/Zenn・Connpass）
- 公開プロフィールページ（`/p/{slug}`）
- ドキュメント整備・インストールガイド
- OSS公開（GitHub）

### 8.3 MVP（Phase 1完了時点）の定義

以下がすべて動けばMVPとして公開できる状態とする。

- エンジニアがスマホから稼働記録を入力・提出できる
- クライアント担当者が稼働報告を確認・差戻しできる
- エージェントが契約を管理できる
- 年間カレンダーで稼働時間の見通しが立てられる
